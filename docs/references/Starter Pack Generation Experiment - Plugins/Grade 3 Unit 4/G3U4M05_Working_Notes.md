## Section Plan

### Warmup (3-5 min)
- **Type:** ACTIVATION — retrieving M4 fact family pattern
- **Prior knowledge activated:** Fact families (M4), commutative pairs within fact families (implicit)
- **Hook:** Display completed fact family (3×5=15, 5×3=15, 15÷3=5, 15÷5=3). Surface the question: "You wrote TWO multiplication equations. They both equal 15. Is that a coincidence — or does this ALWAYS work?"
- **Bridge target:** Creates the question the Lesson answers (why do both multiplication equations give the same answer?)
- **Anticipated interactions:** 2-3 (display + question + bridge)

### Lesson — 3 sections
- **S1 — Commutativity Proof Through Arrays (Early/Concrete-Representational):**
  - Rotation proof with arrays: 3×5 rotated to 5×3, same total
  - Multiple examples (2×7, 4×6) to generalize
  - Student confirms via MC that rotated arrays produce same product
  - "Commutative" introduced informally after visual grounding

## Task 2 Guidance (from Gate 1 Author Review)

### PE-4: S2 Sequencing Principle
Within S2 (Multiplication Tables Grid), explicitly sequence interactions from easier to harder:
1. **First 1–2 partner-finding interactions:** Use 2s/5s (high fluency, low interference per A4). E.g., "Find the commutative partner of 2×5" → student clicks 5×2.
2. **Next 1–2 partner-finding interactions:** Extend to 3s/4s (moderate). E.g., "Find the commutative partner of 3×7."
3. **Diagonal observation:** Arrives AFTER students have found enough partners to see the pattern emerge. Guide-led, not a separate teaching target.
4. **Reverse-direction interactions (highest demand):** Come LAST in S2. "The product is 24 — find the factor pairs." This is the SME addition, and it's the hardest task type.
Source: TVP says "highlight cells students are fluent with first (2s, 5s, 10s) then extend." SP §1.5.2 Guardrails encode this. Make it explicit in interaction ordering.

### CA-3: S3→EC Consolidation Statement
At the S3→EC transition, include an explicit consolidation statement distinguishing the two rules:
- "Order doesn't matter for multiplication — you can switch the factors and get the same product."
- "Order DOES matter for division — switching the numbers gives a different (or impossible) result."
This ensures both properties are verbally crystallized before the Exit Check tests them independently.

## M5 Postmortem

### What went well

**CRA arc held up under scrutiny.** All 12 L2 agents at Gate 4 validated the S1→S2→S3 progression without structural objections. The concrete-to-abstract pipeline (array rotation proofs → table symmetry discovery → division non-commutativity contrast) survived four gates and two full author reviews without requiring arc-level restructuring. The only CRA-related finding across the entire pipeline (CRA3.02 — vocabulary exposure count) was dismissed by the author as sufficient.

**Cross-module coherence was clean from the start.** The M4↔M5 handoff passed 29/29 cross-module checks with zero findings. The deliberate M4 callback value (3×5=15) and vocabulary continuity (fact family → commutative) were designed in at Task 1 and never required correction. The cross-module eval agent confirmed scope boundaries, toy progression, bridge symmetry, and data value continuity all held.

**R1–R9 fix framework worked efficiently.** Jon's consolidated Gate 3 evaluation with a prioritized fix list (Required vs Recommended vs Dismissed) enabled a single focused fix pass instead of iterative back-and-forth. All 9 fixes were applied, verified by L1 re-check, then independently verified by 12 L2 agents at Gate 4. The clear dismiss rationale (with KDD/Decision references) prevented unnecessary rework.

**Gate 3 → Gate 4 delta was small.** The four additional Gate 4 agents (Voice, Cross-Module, Pedagogy, Requirements) all returned PASS with only informational NOTEs. This suggests the SP was already at publication quality after Gate 3 fixes, and Gate 4 primarily served as confirmation rather than surfacing new issues.

### What went wrong

**False positive misclassification cost a full Gate 4 review cycle.** Three L1 checker findings were classified as known false positives in the Gate 3 and Gate 4 evaluation reports when they were actually real issues requiring fixes:

- **ST9 (extra H1):** Classified as "Known FP — task divider" across both reports. Author correctly identified it as a real template violation. The `# WARMUP + LESSON` heading was not in the Structural Skeleton and should never have been introduced. Fix was trivial (delete the line).
- **ST11 (§1.7 ordering):** Classified as "Known FP — G3U4 uses S1/S2/S3 sub-section numbering." Author correctly identified that Required/Forbidden Phrases were in the wrong position per the Structural Skeleton. The checker was right; the FP classification was wrong.
- **ST10 (H4 headings):** Classified as "deliberate organizational choice." Author correctly identified this as an inconsistency with the template. All 13 H4 headings should have been bold inline labels.

**Root cause:** The FP classifications were inherited from earlier modules (M1–M4) that had the same patterns. Because M1 had the same ST9/ST11/ST10 patterns and they were accepted during M1's review, subsequent modules copied the classification without re-examining whether the original decision was correct. This is a form of "FP drift" — an early triage decision propagating forward as established convention.

**Action item:** Review M1–M4 for the same ST9, ST10, and ST11 patterns. At minimum, M4 has confirmed ST11 (Required/Forbidden after Purpose Frame) and ST10 (H4 headings in §1.5 and KDDs).

**VO4 Guide length threshold was too permissive.** The evaluation pipeline treated the VO4 "3 sentences max" threshold as informational, classifying all 15 verbose interactions as MINORs. In reality, the system's interaction model (action steps + "next" button) means that long Guide passages create real UX problems — a student seeing 12 sentences on one screen before they can act is a poor experience. The checker was right to flag these; the severity should have been higher, at least for the extreme cases (7+ sentences).

**Root cause:** The evaluation agents didn't have sufficient context about the delivery system's UX constraints. They evaluated Guide length as a writing quality issue (voice/tone) rather than a product constraint (screen real estate, reading load per step).

**Action item:** Consider promoting VO4 to MAJOR when sentence count exceeds a threshold (e.g., 6+ sentences), or adding a separate check that accounts for the delivery system's step-based interaction model.

**Em dash regression during fix application.** Applying the VO4 trimming edits (G4.4) introduced 7 new VO13 em dashes in Guide text — a regression on a check that had been cleared at Gate 2 (G2.4: 23 em dashes removed). The regression was caught immediately by L1 re-check and fixed in G4.5, but it's a process gap: every Guide text edit should be automatically scanned for VO13 violations before committing.

**Action item:** Consider adding a pre-commit VO13 scan that runs on any edited Guide/Prompt/On Correct text, rather than requiring a full L1 re-check.

**MM0 checker was silently broken for the entire G3U4 pipeline.** The `sp_module_map_check.py` checker has `MODULE_MAP_PATH` and `TVP_PATH` hardcoded to `Grade 3 Unit 2` file paths. For every G3U4 module (M1–M5), the checker has been returning a MINOR "Module not found" and skipping all MM1–MM7 cross-document checks. This means five modules passed through four gates each without any cross-document validation against the Module Map or Toy Flow Plan.

**Root cause:** The checker was originally written for the G3U2 pipeline and the reference paths were never parameterized. The `BASE_DIR` computation (`os.path.dirname(os.path.dirname(SCRIPTS_DIR))`) also resolves to the plugin's `.remote-plugins/` directory, not the workspace — so even if the folder name were corrected, it would look in the wrong location.

**Fix applied:** Patched version created at `sp_module_map_check_patched.py` in the workspace. The patch replaces hardcoded paths with `_discover_reference_files()`, which auto-discovers `*.xlsx` (Module Map) and `*Toy Flow*.docx` files in the SP's own directory. Initial test on M5 confirms it finds both files, resolves M05 in the Module Map, and runs real MM2 vocabulary checks. The patch needs to be applied to the plugin source (`plugin_0127pTNMJD8NG2q7AhCwCXQr/scripts/sp_module_map_check.py`) to take permanent effect.

**Action item:** Apply patch to plugin. Re-run MM checker against M1–M4 to surface any cross-document drift that was missed.
